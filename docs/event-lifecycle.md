# Event Lifecycle

## Event types

| Event        | Generated by          | Meaning                                              |
|--------------|-----------------------|------------------------------------------------------|
| `submitted`  | Intake                | Request received and normalized                      |
| `reviewed`   | Governance            | Request evaluated, decision made (approve/reject)    |
| `validated`  | Validation            | Decision verified against structural rules           |
| `rejected`   | Governance/Validation | Request or decision rejected (with reason)           |
| `executed`   | Execution             | Validated decision implemented                       |
| `correction` | Any authorized role   | Correction recorded as new event, past never mutated |

## Valid transitions

```
submitted ──→ reviewed ──→ validated ──→ executed
                  │              │
                  ↓              ↓
              rejected       rejected
```

| From        | To          | Valid? | Notes                                          |
|-------------|-------------|--------|------------------------------------------------|
| `submitted` | `reviewed`  | ✅     | Every received request must be evaluated       |
| `reviewed`  | `validated` | ✅     | Approved decision moves to structural validation|
| `reviewed`  | `rejected`  | ✅     | Governance can reject at evaluation            |
| `validated` | `rejected`  | ✅     | Validation can reject if rules not met         |
| `validated` | `executed`  | ✅     | Validated decision moves to execution          |
| `submitted` | `executed`  | ❌     | Cannot execute without decision and validation |
| `reviewed`  | `executed`  | ❌     | Cannot execute without structural validation   |
| `executed`  | any         | ❌     | Once executed, the cycle is complete           |

## Core invariant

```
execution requires prior legitimacy
```

Execution only occurs if a complete chain exists:
```
request recorded → decision made → legitimacy verified → execution
```

## Correlation identity (request_id)

Each request receives a unique `request_id` at `submitted`. That identifier propagates to all related events across all pipeline phases, enabling full lineage reconstruction.

## Correction semantics

Corrections do not mutate the ledger. They are recorded as a new event referencing the original:

```json
{
  "event_id": "EVT-0005",
  "request_id": "REQ-0001",
  "action": "correction",
  "corrects": "EVT-0002",
  "reason": "error in actor field"
}
```

This preserves lineage, replay determinism, and historical causality.
